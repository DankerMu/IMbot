# Capability: companion-reconnect

## ADDED Requirements

### Requirement: WSS Auto-Reconnect with Backoff

The companion WSS client SHALL automatically reconnect to the relay when the connection is lost. Reconnection MUST use exponential backoff (1s, 2s, 4s, 8s, 16s, 30s max) with random jitter (0-1s) to prevent reconnect storms.

#### Scenario: relay restarts and companion reconnects

WHEN the relay server restarts
AND the companion's WebSocket connection drops
THEN the companion starts reconnection with 1s + jitter delay
AND on successful reconnect, the companion sends a heartbeat to re-register the host
AND the relay updates the host status to "online"

#### Scenario: companion network blip and recovery

WHEN the companion experiences a brief network interruption
AND the WebSocket connection drops
THEN the companion reconnects using exponential backoff
AND running CLI sessions continue executing locally (they are independent of the relay connection)
AND once reconnected, event forwarding resumes

#### Scenario: backoff with jitter prevents thundering herd

WHEN the relay restarts and multiple companions attempt to reconnect simultaneously
THEN each companion adds a random jitter (0-1000ms) to its backoff interval
AND reconnections are spread over time rather than arriving simultaneously

---

### Requirement: Re-Registration on Reconnect

After a successful reconnection, the companion SHALL immediately send a heartbeat message to re-register the host with the relay. The companion SHALL also report the status of any running CLI processes.

#### Scenario: heartbeat re-registers host

WHEN the companion reconnects to the relay
THEN the companion sends `{ type: "heartbeat", host_id: "<id>", providers: [...], uptime: <n> }`
AND the relay updates `hosts.status = 'online'` and `hosts.last_heartbeat_at`

#### Scenario: running sessions reported on reconnect

WHEN the companion reconnects
AND sessions ["sess-1", "sess-2"] have running CLI processes
THEN the companion sends status updates for each running session
AND the relay can reconcile its session states (e.g., sessions it marked as failed due to disconnect can be restored to running)

---

### Requirement: Event Buffering During Disconnect

The companion SHALL buffer events generated by running CLI processes during a relay disconnect. Buffered events MUST be flushed to the relay on reconnect in order. The buffer MUST be capped at 10,000 events to prevent unbounded memory growth.

#### Scenario: events buffered and flushed on reconnect

WHEN the relay connection drops
AND a running CLI process generates 50 events during the disconnect
THEN all 50 events are buffered in companion memory
AND on reconnect, all 50 events are sent to the relay in original order
AND the relay assigns seq numbers and stores them

#### Scenario: buffer overflow drops oldest events

WHEN the relay connection is down for an extended period
AND buffered events exceed 10,000
THEN the oldest events are dropped to maintain the 10,000 cap
AND a warning is logged indicating N events were dropped

#### Scenario: running CLI processes survive disconnect

WHEN the companion loses connection to the relay
THEN running CLI processes are NOT killed or interrupted
AND their stdout continues to be read and events buffered
AND the user's work is preserved regardless of relay connectivity
